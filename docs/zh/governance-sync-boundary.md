# 治理同步边界

本文说明：从私有 Brain OS 部署向公开 Obsidian Brain OS 仓库同步内容时，什么应该同步，什么不应该同步。

## 一句话规则

> 开源的是可复用的框架能力，不是你的私有运营系统。

同步前先问：

> 一个陌生人 fork 这个仓库后，这个东西对他是可复用的 Brain OS 模式，还是只对我的个人 / 团队环境有意义？

如果是可复用能力，可以考虑同步。如果只是私有胶水，不同步。

## 默认分类

### A 类：主线框架资产 —— 可作为同步候选

满足以下多数条件时，可以考虑同步：

- 直接支持 Brain OS 的知识、治理、自动化主线
- 不依赖你的私有团队上下文也能理解
- 替换占位符后，外部用户可以复用
- 去私有化后安全
- 适合放在 skills、docs、templates、prompts、scripts、schemas 或 cron examples 中

示例：

- 通用知识入库 skill
- vault 治理 skill
- nightly pipeline prompt
- lint / validation 脚本
- 匿名化 cron 示例
- setup / install 文档
- 可复用 schema 与模板

### B 类：边界资产 —— 需要人工确认

满足以下任一条件时，先人工确认：

- 模式有价值，但强绑定你的角色、频道、工具或日程
- 需要大量去私有化
- 容易被误解为官方默认工作流
- 更像内部实现，而不是社区模板

示例：

- 包含可复用 friction schema 的内部史官 prompt
- 有通用计划思想、但深度绑定个人节奏的 personal ops prompt
- 可能值得抽象的内部治理账本

### C 类：私有运营资产 —— 不同步

满足以下任一条件时，不同步：

- 个人日程、日记、提醒、健康、财务、关系上下文
- 私有团队名册、频道 ID、bot ID、webhook、通知路由
- 内部审批流
- 原始日志、原始对话、原始摩擦样本、私有运行报告
- 只用于把你的私有 vault 同步到公开仓库的内部同步器
- 一次性迁移计划或内部项目管理记录

## PII 是最后一层，不是第一层

去私有化是必要条件，但不是充分条件。

一个文件即使不泄露隐私，如果它只服务你的内部运营，也不应该开源。

判断顺序应该是：

1. 它是不是 Brain OS 主线能力？
2. 外部用户是否能复用？
3. 是否能安全去私有化？

## 名称不同是正常的

私有资产名和公开资产名可能不同。

示例：

| 私有概念 | 公开概念 |
|---|---|
| 个人 vault 治理 SOP | `brain-vault-governance` skill |
| 私有同步边界 reference | 公开治理同步边界文档 |

不要因为名字不同就判断为“未同步”。如果你在做私有到公开的同步流程，应该在私有项目记录中维护一张同步登记表。

## 推荐维护同步登记表

如果你长期从私有 Brain OS 部署向公开仓库同步改进，建议维护类似表格：

```markdown
| 私有资产 | 公开资产 | 状态 | 最近同步 | 备注 |
|---|---|---|---|---|
| `~/.agents/skills/my-vault-sop/` | `skills/brain-vault-governance/` | 已同步 | 2026-04-26 | 公开抽象版 |
```

这能防止后续 agent 把已经同步过的资产重复判断为新候选。

## 私有内容应该抽象成什么

| 私有内容 | 应该发布成 |
|---|---|
| 原始团队 prompt | 匿名化 prompt 模板 |
| 私有运营规则 | 可复用治理原则 |
| 个人日程 cron | 带占位符的通用 cron 示例 |
| 原始运行报告 | 报告模板或 schema |
| 内部迁移计划 | 稳定的用户文档 |
| 写死路径的私有脚本 | 参数化脚本 + 配置示例 |

## 同步前最小 checklist

- [ ] 这是可复用的 Brain OS 能力。
- [ ] 私有人名、路径、ID、webhook、项目细节已移除。
- [ ] 用户个人节奏已泛化，或只作为示例存在。
- [ ] 不包含原始日志和原始证据。
- [ ] 占位符已有说明。
- [ ] 公开仓库中有清晰落点。
- [ ] 必要时更新 changelog 或 release notes。

## 保守默认

不确定时，不要同步原始资产。

更好的做法是先写一个简短计划或 issue，说明：

- 哪部分可能可复用
- 哪部分是私有的
- 需要怎样抽象
- 需要谁来确认
