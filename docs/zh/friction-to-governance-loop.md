# Friction-to-Governance Loop

Brain OS 不应该只帮助创造知识和运行工作流。随着系统变复杂，它也应该帮助减少重复犯错。

这份文档说明一种可复用的模式：如何把反复出现的摩擦信号，转化成治理升级。

---

## 这条 loop 是什么

friction-to-governance loop 的核心很简单：

1. 收集摩擦信号
2. 把它们归成重复问题类型
3. 找到真正的根因
4. 把修复回写进系统

目标不是为了“反思而反思”。

目标是让 Brain OS 随时间推移变得更稳。

---

## 什么算 friction

friction signal 指的是重复出现或带有结构性的麻烦，例如：
- agent 总把文件写到错误位置
- 同一条规则一再被忘记
- 某个 cron 或 workflow 总扫不到真正重要的升级面
- prompt 技术上没错，但在执行上不断误导
- 用户必须反复纠正同一种行为

一次性的错误不够。

这条 loop 面向的是反复出现的模式。

---

## 四个阶段

### 1. Capture
收集重复摩擦的具体证据：
- 用户纠正
- 错误或误导性输出
- 工作流反复漏检
- 重复的计划或治理错误

### 2. Diagnose
判断真正坏的是哪一层：
- prompt 问题
- AGENTS / 团队规则问题
- reference 问题
- 扫描 / 自动化逻辑问题
- 命名或结构问题

### 3. Govern
选择正确的治理动作：
- 重写 prompt
- 强化 AGENTS 提示
- 补 reference 或改 reference
- 调整扫描规则
- 改默认 rollout 路径

### 4. Write back
只有当修复被写回系统，这次修复才算完成。

例如：
- 更新 `AGENTS.md`
- 更新某个 reference 文件
- 更新 onboarding 文档
- 更新某个 cron prompt
- 更新 release guidance

---

## 它不是什么

它不是：
- 私人日记
- 抽象自我反思
- 一个把所有错误都堆进去的地方
- 一个要求把每个小问题都系统化的机制

这条 loop 应该是有选择的。

它处理的是重复摩擦，不是日常噪音。

---

## 为什么它重要

如果没有治理 loop，Brain OS 可能会不断增加功能，但重复犯同样的错。

有了治理 loop，重复摩擦就能变成系统升级的来源。

这会让系统更可教、更稳定，也更容易被别人采用。

---

## 常见回写目标

摩擦信号通常会回写到这些层之一：

- **Prompt 层**：执行说明不清时
- **Reference 层**：规则存在，但很容易被跳过或忘记时
- **AGENTS 层**：规则必须在行动前可见时
- **Workflow 层**：自动化或扫描逻辑没有抓住真正问题时
- **Docs / onboarding 层**：外部用户或 agent 还无法正确理解新能力怎么用时

---

## 一个典型例子

某个重复问题出现：
- formal plan 反复被写进 workspace，而不是知识库

诊断：
- 计划治理规则本身存在，但 reference 没有足够早地被读取

治理动作：
- 强化 `AGENTS.md`
- 强化相关 reference
- 强化对应 cron / sync prompt

回写结果：
- 后续运行更容易把正式计划写到正确位置

---

## 推荐采用顺序

对 OSS 用户来说，这条 loop 应该分阶段采用：

1. 先让一个工作流成功
2. 再识别重复摩擦
3. 再定义一个小的治理 loop
4. 最后再加模板、报告或自动化

不要一开始就上一个很重的内部治理栈。

先学会这个模式，再扩展。

---

## 它和 Brain OS 其他部分的关系

- **Knowledge governance** 维护的是知识库健康
- **Friction-to-governance** 维护的是系统运行健康
- **Observer** 或类似 review agent 可以提供输入
- **References** 和 **AGENTS** 是常见回写目标

这些层是互相强化的。

---

## 最后原则

> 重复摩擦不应该一直只是重复摩擦。
> 它应该变成治理。
