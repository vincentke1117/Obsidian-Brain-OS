# Brain Vault Governance

## Purpose

This skill keeps a long-lived Brain OS vault from drifting into duplicated truth sources, misplaced plans, broken links, and asset dumps.

Use it before changing vault structure or writing durable files.

## Core Rules

1. **Inspect before writing** — never place files from memory.
2. **Plans and prompts are different artifacts** — plans are for human review; prompts are for agents to execute.
3. **The vault is the durable source of truth** — workspaces may hold drafts, not long-term official plans.
4. **Move notes with vault-aware tools** — avoid breaking Obsidian links.
5. **Git stores conclusions; local storage keeps raw evidence** — do not commit bulky raw capture assets.
6. **Prefer reuse over new structure** — attach to existing folders before creating new ones.

## When to Use

Use this skill before:

- creating or updating formal project plans
- deciding where a document belongs
- adding folders or numbered sections
- moving or renaming notes
- converting workspace drafts into vault documents
- writing task/todo records
- ingesting external articles or raw assets
- auditing duplicated or misplaced files

Do not use it for quick scratch notes that will not become durable vault records.

## Required Preflight

Before writing a durable file:

1. Identify artifact type:
   - plan
   - prompt
   - knowledge note
   - project document
   - task/todo
   - raw asset
   - report/log
2. Inspect the target parent directory:
   ```bash
   find "{{BRAIN_ROOT}}/<target-parent>" -maxdepth 1 -type d | sort
   find "{{BRAIN_ROOT}}/<target-parent>" -maxdepth 1 -type f | sort | head -30
   ```
3. Read at least 2-3 nearby files when the directory's intent is unclear.
4. Decide whether to reuse an existing directory or create a new one.
5. If creating a directory, verify the numbering scheme at that exact level.

## Placement Rules

| Artifact | Default home |
|---|---|
| Formal project plan | `{{BRAIN_ROOT}}/05-PROJECTS/<project>/plans/` |
| Project sprint / implementation plan | `{{BRAIN_ROOT}}/05-PROJECTS/<project>/plans/sprints/` if that structure exists |
| Project insight proposal | `{{BRAIN_ROOT}}/05-PROJECTS/<project>/plans/insights/` if generated by an insight pipeline |
| Cross-project governance plan | `{{BRAIN_ROOT}}/01-PERSONAL-OPS/06-AGENT-PLANS/` or your vault's governance area |
| Agent prompt / cron prompt | `{{BRAIN_ROOT}}/04-SYSTEM/prompts/` or the repository's prompt area |
| Task/todo backlog item | `{{BRAIN_ROOT}}/00-INBOX/todo-backlog.md` or your configured task inbox |
| Knowledge note | `{{BRAIN_ROOT}}/03-KNOWLEDGE/` under the appropriate domain/system layer |
| External raw assets | `{{BRAIN_ROOT}}/LOCAL-LARGE-FILES/knowledge-sources/<domain>/<yyyy-mm-dd>-<slug>/` |
| Run report / integration report | `{{BRAIN_ROOT}}/03-KNOWLEDGE/99-SYSTEM/` or your configured reports area |

These are defaults. If your vault template defines a different structure, follow the local README or index in that area.

## Plan vs Prompt Decision

Ask:

> Is this document for a human to review/coordinate/approve later, or for an agent to execute directly?

- Human review, coordination, approval, retrospective → **Plan**
- Agent execution instructions, cron body, reusable automation prompt → **Prompt**
- Unstable thought surface → **Draft**

A formal plan must not live indefinitely in a workspace or prompt folder.

## Folder Governance

Before creating a folder:

1. Check the real current structure.
2. Check numbering only at the current sibling level.
3. Prefer existing parents over new top-level areas.
4. Avoid duplicate semantic homes.
5. Treat top-level numbered folders as high-risk changes.

High-risk changes require human confirmation:

- new top-level numbered directories
- renaming top-level directories
- bulk migrations across major vault areas
- changing global governance conventions

## Moving and Renaming Notes

If the vault has a link-aware tool such as `obsidian-cli`, use it for note moves and renames:

```bash
obsidian-cli move "old/path" "new/path"
```

Avoid raw `mv` for Markdown notes when backlinks or wikilinks may exist.

Direct file editing is fine for content-only changes.

## Raw Asset Boundary

Git should contain the knowledge conclusion layer:

- distilled notes
- topic pages
- patterns
- reports
- lightweight metadata

Local large-file storage should contain raw evidence:

- screenshots
- OCR image packs
- PDFs and attachments
- debug HTML
- API responses
- crawler outputs

Default local raw asset layout:

```text
LOCAL-LARGE-FILES/knowledge-sources/<domain>/<yyyy-mm-dd>-<slug>/
```

## Task vs Todo Rule

Create a **task** when any of these are true:

- spans more than one day
- requires multiple people/agents
- has a formal deliverable
- needs progress/risk/ETA tracking

Otherwise create a **todo**.

Todos should stay in a single configured backlog/inbox, not scattered across project notes.

## Completion Standard

After changing the vault:

1. Run the smallest useful verification:
   - inspect file path
   - run relevant lint/check script
   - verify links if moved
   - check git status
2. Commit durable vault changes when the vault is git-managed.
3. Report:
   - files changed
   - commit hash, if committed
   - clean/dirty status

## Common Mistakes

| Mistake | Fix |
|---|---|
| Writing formal plans to a workspace forever | Promote them to the project/governance plan area |
| Putting agent prompts into plan folders | Move prompts to the prompt area |
| Creating folders from memory | Inspect real structure first |
| Reusing top-level numbers incorrectly | Check numbering at the sibling level |
| Committing raw capture dumps | Store raw assets under local large-file storage |
| Moving Markdown notes with raw `mv` | Use a link-aware move tool when available |
